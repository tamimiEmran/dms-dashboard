DMS Onboarding

PyTorch pipeline · driver monitoring · for new contributors

DMS is a pipeline, not a project with one model. Every experiment has the same shape. Five stages, each pluggable, each registered by string name, each configured by YAML. Note that shape and 80% of the codebase maps itself.

Where to go from here
Canonical reading order
From this page through to the source code — fastest path to being productive.
  1. Read this page end-to-end. Pay attention to the pipeline diagram — that shape repeats.
  2. CONTRIBUTING.md — dev setup, the three coding rules, PR conventions. ~200 lines.
    Enforced in review; skipping it produces PRs that bounce.
  3. CLAUDE.md §4 (Module Map) and §5 (Architecture). ~50 lines total.
    Labelled "for AI assistants" but higher-signal than any human doc. Not optional.
  4. The pipeline deck (link above). Stop at whichever stage your first task touches.
  5. The per-subsystem CLAUDE.md for that stage: src/dms/data/CLAUDE.md, src/dms/models/CLAUDE.md, src/dms/training/CLAUDE.md, src/dms/evaluation/CLAUDE.md.
  6. Source code. Start from src/dms/data/datamodule.py or src/dms/training/trainer.py — whichever matches your task. Both are shortest-path into real behaviour.
