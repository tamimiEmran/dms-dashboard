Configuration

Hydra + Registry

Every knob is a YAML value. Every string resolves to a class via a registry. Combining them = your experiment.

9 Config groups
13 Registries

Two systems, one loop. Hydra composes YAML groups at load time so every run writes a fully resolved config to experiments/runs/<name>/.hydra/. Registries resolve strings to classes at import time so a name in YAML (name: resnet50) finds the right Python class.

configs/ tree

configs/
├── config.yaml        ← root: default group selection
├── dataset/           ← 11 YAMLs
├── model/             ← 8 YAMLs
├── trainer/           ← 7 YAMLs (6 fine-tune strategies)
├── augmentation/      ← 3 YAMLs (light / medium / heavy)
├── evaluation/        ← 4 YAMLs
├── compression/       ← 5 YAMLs
├── inference/         ← 3 YAMLs
├── assessment/        ← 1 YAML
└── experiment/        ← multi-group overrides

What Hydra does

  • Reads configs/config.yaml and follows the defaults: list.
  • Picks one file from each group. Produces one merged config tree.
  • CLI overrides mutate that tree before code runs.
  • Writes the final tree into .hydra/ next to the checkpoint.

What the registry does

  • 13 registries in dms.registry. One per extension type.
  • Each registry is a dict: string name → class.
  • A class joins by decorator: @metric_registry.register("my_metric").
  • Registration fires when the module is imported.
If you can write a YAML file, you can run an experiment. No code changes required.

configs/config.yaml

# configs/config.yaml
defaults:
  - dataset: dmd
  - model: mobilenetv3_large
  - trainer: default
  - augmentation: light
  - evaluation: default

How one file per group is picked

  • Each line under defaults: names a group and a file stem.
  • Stem dmd means load configs/dataset/dmd.yaml.
  • Merged top-down. Later entries win on key collisions.
  • CLI overrides run after the merge; they touch the merged tree.
Output directory template. Runs land at experiments/runs/{model.name}/. The .hydra/ subfolder inside each run is the full composed config — use it to reproduce experiments months later.

Overrides / 02

key= / key.sub= / +key=

Three forms. Silently different results.

Form 1
Swap group
key=value 
python scripts/training/train.py model=resnet50
Effect: loads configs/model/resnet50.yaml instead of the default mobilenetv3_large.yaml. Whole group replaced.
Form 2
Edit leaf
key.sub=value 
python scripts/training/train.py trainer.epochs=10 trainer.optimizer.lr=3e-4
Effect: still uses the default trainer/default.yaml, but overrides two leaf values inside.
Form 3
Add key
+key=value 
python scripts/training/train.py +trainer.my_new_flag=true
Effect: adds a new key that did not exist in the YAML. Without +, Hydra raises on unknown keys — that is the safety. The + waives it.

Combined example

python scripts/training/train.py \
    model=resnet50 \
    dataset=statefarm \
    trainer=finetune_lora \
    trainer.epochs=10 trainer.optimizer.lr=3e-4 \
    +trainer.log_every=50
Caveat. Using the wrong form does not error — it just runs a config that is subtly different from what you meant. Bad runs blamed on bugs are often one of these three.

Registries / 03

Subclass → decorate → import → reference

Four steps. Every extensible thing in DMS works the same way.

  1. Subclass the base: class MyMetric(BaseMetric):
    Required class attrs enforced via __init_subclass__.
  2. Decorate: @metric_registry.register("my_metric")
    The string key is what YAML references.
  3. Import: add to src/dms/evaluation/metrics/__init__.py
    Registration fires on import. No import = no entry.
  4. Reference: put "my_metric" in a YAML or CLI override.
    Hydra then pulls the class back out of the registry by name.

Full example — a metric

from dms.evaluation.metrics.base import BaseMetric, MetricResult
from dms.registry import metric_registry


@metric_registry.register("precision_at_recall")
class PrecisionAtRecall(BaseMetric):
    name = "precision_at_recall"
    requires = ["predicted_labels", "true_labels"]
    higher_is_better = True

    def compute(self, predictions) -> MetricResult:
        ...

    def format(self, result) -> dict:
        ...
