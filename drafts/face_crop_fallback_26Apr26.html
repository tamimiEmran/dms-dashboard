TL;DR

  • The CropSpec declarations are correct: every face-trained model asks for FACE; every distractedness/full-frame model asks for FULL.
  • The exposure is the silent fallback in multi_pipeline.py:85-90 — when the shared FaceCropExtractor finds no face, CropSpec.FACE resolves to the full frame, and the classifier inferences proceed without any flag in the leaderboard CSV.
  • For the 178 fallback frames, drowsiness ViTs / SigLIP2 / MobileViT models received a 1920×1080 cabin frame. Some respond with low-confidence (chbh7051 ViT), some respond with very high confidence (mosesb_mobilevitv2, doze_guard) — both directions distort the per-class metrics on the leaderboard.

Three independent face-detection signals, three different answers

The shared FaceCropExtractor (used by every CropSpec.FACE consumer) doesn't directly expose its success rate, so we triangulate three signals from the wide.csv files:

Signal What it measures Source Failure rate
gaze__face_available Crop received by gaze detector was ≤512 px (heuristic for "real face crop, not fallback frame") gaze_detector.py:260 28.4%
eye_state__face_detected MediaPipe FaceMesh succeeded on the full cabin frame eye_state_detector.py internal 27.5%
mediapipe__meta__face_detected MediaPipe FaceMesh succeeded on the full cabin frame (separate detector instance) mediapipe_detector.py 27.5%

Headline rates agree (~28%) but the signals disagree on which frames fail: only 7.5% of frames have all three signals failing simultaneously, and 48.4% of frames have at least one signal failing. The FaceCropExtractor (gaze proxy) and MediaPipe FaceMesh (eye_state & mediapipe) are genuinely different detectors and miss different subsets of the frames.

For the leaderboard: only the FaceCropExtractor result matters — it's the one that decides what crop a face-trained classifier receives. Hence the gaze size-heuristic is used as the proxy below.

Root cause — the silent fallback in CropProvider

src/dms/inference/multi_pipeline.py:85-90: 

if spec is CropSpec.FACE:
    if not faces:
        crop = self._frame      # ← silent fallback to full frame
    else:
        crop = faces[0][0]

The fallback is intentional — it preserves the "every detector returns something for every frame" contract, which keeps downstream fusion simple. But the fallback flag stays inside per-detector metadata (head_pose__face_available, gaze__face_available, etc.), and the scoring layer doesn't read it. Predictions made on the fallback frame are scored identically to face-cropped predictions.

The asymmetry per detector is also worth flagging:

  • head_pose always reports face_available = 1.0 (line 46-48 of head_pose_detector.py: “the detector itself has no way to tell a face crop from a fallback frame — the pipeline's own face-detection metadata is the source of truth”).
  • gaze reports face_available via a size heuristic (max(h,w) ≤ 512) — useful proxy because Hikvision frames are always >512.
  • classifier_* detectors don't surface face_available at all in their wide.csv. They blindly inferred without the fallback flag making it into the leaderboard.

Recommended fixes

Short-term — rescore

Re-pivot the leaderboard with a face-only filter for CropSpec.FACE models: 

df = pd.merge(model_tidy,
              gaze_proxy[['sample_id', 'face_avail']],
              on='sample_id')
df = df[df.face_avail > 0]    # face-only
metric = compute(df.preds, df.gt)

No new inference required — the wide.csv files already contain everything; only the metric pivot changes.

Medium-term — surface the flag

Add face_available to the BaseDetector contract, populated by CropProvider when it hands a CropSpec.FACE consumer the fallback frame. Then either:

  • the metric layer drops face_available == False rows for face-trained models, or
  • the pipeline returns None for those frames (matching how MediaPipe handles “no face” today).

Either path makes the no-silent-failure rule from CLAUDE.md apply at the metric layer too, not just at the detector layer.

Independently — investigate the saturated rows

The three teen_d_* classifiers always emit prob = 1.0. Likely root causes (in priority order):

  • preprocessing mismatch — the upstream training pipeline used a normalization the wrapper isn't applying.
  • checkpoint head transplant issue — the head-preservation fix in build_composed_from_pretrained (CLAUDE.md §8 item 31) may have edge-cased these specific weights.
  • label-mapping inversion — the binary head's class indices are swapped vs the registered class_mapping.

Why this matters for OEM deployment

On OEM cabin frames (which we expect to also have ~28% face-detection failures during night / IR / occlusion conditions), a fatigue model that silently fires at 95% on full-cabin frames will drown the alert system in false positives. The Hikvision audit gives us the early read on which candidate models are robust to face-detection failures and which are fragile.

Appendix — methodology

How fallback frames were identified

  1. Read the latest gaze_l2cs_net__*__wide.csv from outputs/leaderboards/hikvision/predictions/ (run 20260420T161120Z, n=626).
  2. Use gaze__face_available as the per-frame proxy: 1.0 means the detector received a crop ≤512 px (real face); 0.0 means it received the full frame (fallback). This is the size heuristic in gaze_detector.py:260.
  3. For each CropSpec.FACE model, read the latest run's wide.csv and inner-join on sample_id. Verified all 10 models share identical 448 face / 178 fallback splits — expected because they all consume the same shared CropProvider cache.
  4. Per canonical class, compute mean of prob column and fires rate stratified by face_available.
  5. Cross-validate with two independent FaceMesh signals (eye_state__face_detected, mediapipe__meta__face_detected); agreement is on the headline rate (~28%) but not on which frames fail.

Caveats

  • The gaze size-heuristic catches the cache's fallback path correctly but won't catch failures where the cache returned a small crop that still wasn't a real face. Without the ground-truth face-detection log, this is the best in-band proxy.
  • Per-frame ground truth for “is this driver actually fatigued” isn't joined here — that requires the GT loader and is out of scope for this audit. The metrics on the leaderboard remain valid as configured; what this audit shows is that the leaderboard is mixing two distinct evaluation regimes (face vs fallback) without distinguishing them.
  • The proxy run is from 2026-04-20 16:11 UTC; classifier wide.csv files used in the per-class numbers are also from 2026-04-20 (see run_id per row). Frame-set parity is guaranteed because the leaderboard fixes HIKVISION_MANIFEST globally.
