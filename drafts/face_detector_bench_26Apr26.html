Three things this bench changes

Production was on Haar all along

FaceCropExtractor tries to load res10_300x300_ssd_iter_140000.caffemodel from the OpenCV install dir. Those weights ship with some opencv-contrib builds but not ours. The class falls back to Haar without raising — the logger.info is the only signal. So the 28.4% Hikvision fallback rate measured earlier is a Haar-cascade-on-cabin-frames result, not the 2014 SSD model. A modern detector lifts recall by ~58 percentage points.

The bootstrap pipeline already validated this

yolov12l_face (which the face/person bootstrap experiment uses) lands at F1=0.955 here too — consistent with the OEM bootstrap result (23.1% → 2.2% missed-face rate). Either of buffalo_l / yolov11n_face / yolov12l_face would do. buffalo_l wins on perfect precision; yolov12l_face wins on max recall but has 5 FPs to buffalo_l's 0.

MediaPipe FaceMesh is the wrong tool for face detection

eye_state and mediapipe detectors use FaceMesh internally for face localization. FaceMesh's recall is 35.7% on cabin frames — only marginally better than Haar. The eye-closed and EAR signals these detectors emit are accurate when a face is located, but they're effectively missing 64% of the time. Worth wiring them to the same upstream face detector instead of running their own FaceMesh per frame.

Speed is not the bottleneck

Best F1 (buffalo_l) runs at 7 ms / frame on a desktop GPU; SCRFD-2.5G at 5 ms. For a 30 fps pipeline that's 15–21% of the frame budget — well within real-time. Even on a Jetson Orin Nano (~3-5× slower depending on TensorRT), we're at 25–35 ms, still <1 frame at 30 fps. The Haar cascade is actually the slowest in the bench at 31 ms, for a 60-pt-worse F1.

Next steps

  1. Wire FaceCropExtractor to use insightface. The class already supports this (line 5 of face_crop.py). One-line config change at construction site in multi_pipeline.py.
  2. Re-run the leaderboard. Expect the 28.4% fallback rate measured earlier to drop to single digits. Per the per-model contamination audit, this should fix the worst contamination (mosesb_mobilevitv2's 6× gap) by handing those 178 frames real face crops instead of full cabin frames.
  3. Wire eye_state, head_pose, gaze, mediapipe to share the new face crops. These detectors currently run their own FaceMesh / no-detection logic. With a shared detector at 92% recall, their effective recall ceiling lifts too — and we save the per-frame redundant face-detection passes.
  4. Defer driver-vs-passenger filtering to a separate step. The bootstrap module's derive_face_category handles this via center-in-box containment. Apply it after detection but before the classifier inputs.

Methodology

  • GT: 54 Hikvision frames hand-reviewed in the face/person bootstrap pipeline; 112 face boxes in COCO format at experiments/face_person_validation/hikvision/annotations.coco.json.
  • Matching: greedy by descending confidence, IoU≥0.5 with each GT face. Each GT can match at most one prediction; unmatched predictions are FPs, unmatched GTs are FNs.
  • Inference: single image at a time, no batching, GPU when available (RTX 3060 Ti). Mean time excludes the warm-up frame.
  • Default thresholds: each detector at its package default (or conf=0.25 for ultralytics YOLO). Tuning per detector could shift the precision/recall tradeoff but not the F1 ranking.
  • Source artifacts: outputs/leaderboards/hikvision_face_detector_bench.csv (per-frame), outputs/leaderboards/hikvision_face_detector_bench_summary.csv (per-detector aggregates).
